📡 Niye channel? Why a channel?

🏃━━━🏃

iki goroutine bir boruyla konuşur two goroutines talk through a pipe

"Belleği paylaşma. Mesaj yolla." "Don't share memory. Pass messages."

Channel = goroutine'ler arası tip-güvenli boru. Bir tarafa veri koyarsın, diğer taraftan biri çeker. A channel = a typed pipe between goroutines. One side sends, the other receives.

Nasıl yarıyoruz? How do we make one?

2/11

🛤️ make ile yarat Create with make 

ch := make(chan int)         // int taşıyan kanal

mesajlar := make(chan string)   // string taşır

make(chan T) — T tipinde değer taşıyan boru. make(chan T) — a pipe carrying values of type T.

Tipi sabittir: chan int'e sadece int gönderirsin. Karışık tip istersen chan any. Type is fixed: only ints into chan int. Mixed types need chan any.

Nasıl gönderir/alırsın? How to send and receive?

3/11

📨 <- ile gönder ve al <- to send and receive 

ch <- 42           // gönder: ok <- veri

x := <-ch          // al: değişken := <-kanal

<-ch               // al ama değeri at

Ok yönü = veri akışı. Sol taraf alıcı, sağ taraf kaynak. Arrow = data flow. Left receives, right is source.

Tek operatör (<-) iki yönü temsil eder. Konum kim konuştuğunu söyler. One operator (<-) covers both directions. Position tells who's speaking.

Tampon yoksa ne olur? What if there's no buffer?

4/11

🤝 Tamponsuz — el sıkışma Unbuffered — handshake 

ch := make(chan int)

go func() {
    ch <- 42     // alıcı gelene kadar bekler
}()

x := <-ch       // gönderen olana kadar bekler

// İkisi buluştuğunda iş geçer.

Gönderen ve alıcı aynı anda hazır olmadan iş geçmez. Sender and receiver must be ready at the same time for the transfer to happen.

Bu özellik zamanlama (timing) sorunlarını çözer. WaitGroup yerine bazen channel kullanırsın. This solves timing problems naturally. Sometimes you'll use a channel instead of WaitGroup.

Beklemeden gönderme? Sending without waiting?

5/11

📦 Tamponlu kanal Buffered channel 

ch := make(chan int, 3)   // 3 kapasite

ch <- 1   // dolu değil → bekleme yok
ch <- 2
ch <- 3
ch <- 4   // dolu → blok

İkinci argüman = kuyruk boyutu. Dolana kadar gönderim engellenmez. Second argument = queue size. Sends don't block until it's full.

Doluyken yazmak ya da boşken okumak yine bekler. Tampon performans için, sürpriz için değil. Writing when full or reading when empty still blocks. Buffers are for throughput, not surprise.

"Bittim" demek? Saying "I'm done"?

6/11

🔚 close + range close + range 

go func() {
    for i := 1; i <= 3; i++ {
        ch <- i
    }
    close(ch)         // "artık veri yok"
}()

for v := range ch {    // kapanana kadar gez
    fmt.Println(v)
}

Gönderim biterse close. range kapanmayı görünce durur. When sending ends, close. range stops when it sees the close.

Sadece gönderen kapatır. Kapalı kanala yazmak panic. Tekrar kapatmak panic. Only the sender closes. Writing to a closed channel panics. Closing twice panics.

Birden fazla kanalı dinlemek? Listening to multiple channels?

7/11

🎰 select — hangi gelirse select — whichever arrives 

select {
case v := <-ch1:
    fmt.Println("ch1:", v)
case v := <-ch2:
    fmt.Println("ch2:", v)
case <-time.After(1*time.Second):
    fmt.Println("timeout")
}

Birden fazla kanal arasında ilk hazır olanı seçer. Picks the first ready channel among many.

İkisi aynı anda hazırsa rastgele biri çalışır. default case eklersen hiç bekleme yapmaz. If multiple are ready, one is picked at random. Add default to skip waiting entirely.

Kanalın yönünü kısıtlamak? Restricting direction?

8/11

↗️ Yön kısıtlaması Direction restriction 

// Sadece gönderebilen function:
func uretici(ch chan<- int) {
    ch <- 42
}

// Sadece alabilen function:
func tuketici(ch <-chan int) {
    fmt.Println(<-ch)
}

chan<- T sadece gönder · <-chan T sadece al. chan<- T send only · <-chan T receive only.

Function imzasında kullanırsan derleyici kanalı yanlış yönde kullanmanı engeller. In function signatures, the compiler stops you from using the channel the wrong way.

En sık tuzaklar? Common traps?

9/11

🪤 Tuzaklar Traps

close edilmiş'e yaz→ panic
iki kez close→ panic
kimse almıyor→ deadlock (her iki taraf bekliyor)
nil channel→ sonsuza kadar bloklar
write to closed→ panic
close twice→ panic
no receiver→ deadlock (everyone waits)
nil channel→ blocks forever

Channel güçlü ama hassas — kuralları unutma. Channels are powerful but picky — keep the rules in mind.

Pratik kural: "Gönderen kapatır." "Alıcı close olduğunu range veya v, ok := ile öğrenir." Practical rule: "Sender closes." "Receiver detects close via range or v, ok :=."

Çok kullanılan kalıp? A common pattern?

10/11

👥 Worker pool kalıbı Worker pool pattern 

isler   := make(chan int, 10)
sonuc := make(chan int, 10)

// 3 worker başlat
for w := 0; w < 3; w++ {
    go func() {
        for n := range isler {
            sonuc <- n * n
        }
    }()
}

for i := 1; i <= 5; i++ { isler <- i }
close(isler)

Bir kuyruk + N worker = kontrollü paralelizm. One queue + N workers = controlled parallelism.

Çok yapılan iş için en yaygın Go kalıbı. Kapasiteyi kontrol edersin, sistemi boğmazsın. The most common Go pattern for batch work. You control parallelism — don't melt the box.
