🚦 Niye mutex? Why a mutex?

🏃⛔🏃

aynı veriye iki goroutine — bir seferde bir tane içeri same data, two goroutines — one at a time

Mutex = "tek bir goroutine girebilir" kapısı. A mutex = a door only one goroutine can pass.

"Mutex" = mutual exclusion (karşılıklı dışlama). Channel mesaj yollar; mutex doğrudan veri korur. "Mutex" = mutual exclusion. Channels pass messages; a mutex protects shared state directly.

Lock ve Unlock? Lock and Unlock?

2/10

🔒 sync.Mutex — Lock/Unlock sync.Mutex — Lock/Unlock 

var mu sync.Mutex
sayac := 0

go func() {
    mu.Lock()
    sayac++
    mu.Unlock()
}()

Lock() kapıyı kapatır, Unlock() açar. Lock() closes the door, Unlock() opens it.

var mu sync.Mutex sıfır değeriyle hazır — make gerekmez. Diğer goroutine'ler kilit boşalana kadar bekler. var mu sync.Mutex is ready at zero value — no make needed. Other goroutines wait until it's free.

Unlock'u unutmamak? Not forgetting Unlock?

3/10

⏰ defer ile garanti Guarantee with defer 

func artir() {
    mu.Lock()
    defer mu.Unlock()    // her durumda çalışır

    sayac++
    // panic olursa bile Unlock garantili
}

defer mu.Unlock() — Lock'tan hemen sonra yaz. defer mu.Unlock() — write it right after Lock.

Function nasıl biterse bitsin (return, panic) Unlock çalışır. Unutulmuş Unlock = donmuş program. However the function ends (return, panic), Unlock fires. A forgotten Unlock = a frozen program.

Çoğu okuma, az yazma? Mostly reads, few writes?

4/10

📚 RWMutex — birçok okur RWMutex — many readers 

var mu sync.RWMutex

// Çok okur paralel — RLock
mu.RLock()
v := veri[anahtar]
mu.RUnlock()

// Yazar tek başına — Lock
mu.Lock()
veri[anahtar] = 42
mu.Unlock()

Birçok okur paralel girer. Bir yazar herkesi bekletir. Many readers run together. One writer blocks everyone.

"Okuma çok, yazma az" senaryolarında daha hızlı. Aksi durumda sync.Mutex'in kendisi yeterli. Faster when reads dominate. Otherwise, plain sync.Mutex is fine.

Goroutine'ler bittiğinde? When goroutines finish?

5/10

🔢 WaitGroup — recap WaitGroup — recap 

var wg sync.WaitGroup

for _, url := range urlList {
    wg.Add(1)
    go func() {
        defer wg.Done()
        indir(url)
    }()
}
wg.Wait()

N goroutine'in bitmesini beklemek için ideal araç. The right tool for "wait until N goroutines finish".

Add'i her zaman go'dan önce çağır. Goroutine içinde Add çağırırsan yarış olur. Call Add before the go. Calling Add inside the goroutine creates a race.

Bir kez yapılması gereken? One-time setup?

6/10

1️⃣ sync.Once — bir kez sync.Once — exactly once 

var once sync.Once
var config *Config

func getConfig() *Config {
    once.Do(func() {
        config = loadConfig()    // sadece bir kez
    })
    return config
}

once.Do(fn) — kaç kez çağırılırsa çağrılsın fn bir kez çalışır. once.Do(fn) — no matter how many calls, fn runs once.

Tek seferlik kurulum (singleton, kapasitesi yüksek hesap, vs.) için temiz çözüm. Clean way to do one-time setup (singletons, expensive init, etc).

Daha hızlı, daha küçük? Faster, smaller?

7/10

⚛️ atomic — kilitsiz sayaç atomic — lockless counter 

import "sync/atomic"

var sayac int64

go func() {
    atomic.AddInt64(&sayac, 1)
}()

n := atomic.LoadInt64(&sayac)

Tek bir int için kilit fazla. atomic CPU seviyesinde yapar. A lock for a single int is overkill. atomic does it at CPU level.

Sadece sayaçlar, flag'ler gibi tek bir değer. Karmaşık veriyi koruyacaksan mutex daha uygun. Only for single values — counters, flags. For richer data, prefer a mutex.

Mutex mi channel mı? Mutex or channel?

8/10

🤔 Mutex mi, channel mı? Mutex or channel?

durum koruma→ mutex (counter, cache, registry)
veri akışı→ channel (pipeline, queue, fan-out)
sahiplik geçişi→ channel ("ben ürettim, sen al")
protect state→ mutex (counter, cache, registry)
data flow→ channel (pipeline, queue, fan-out)
handing off→ channel ("I made it, you take it")

Sabit veriyi koru → mutex. Veri akıt → channel. Protecting state → mutex. Moving data → channel.

"Birini diğerinden yapay olarak çıkarma." İhtiyacın hangisini gerektiriyorsa onu kullan. "Don't force-fit one into the other." Use whichever your problem actually needs.

Sık tuzaklar? Common traps?

9/10

🪤 Tuzaklar Traps

Mutex'i kopyala→ struct'ta mutex varsa pointer ile geç (*T)
Unlock unutmak→ defer kullan, daima
İç içe Lock→ deadlock; aynı kilitle ikinci kez girersen donar
Lock olmadan oku→ race; -race onu yakalar
Copying a mutex→ pass *T if struct contains a mutex
Forgetting Unlock→ always use defer
Re-entrant Lock→ deadlock; same lock can't enter twice
Read without Lock→ race; -race finds it

Mutex hassas bir araç. Kuralları ihmal etme — donar. A mutex is a sharp tool. Bend the rules and the program freezes.

"Mutex'i mümkün olan en küçük bölgeyi kapsayacak şekilde kullan." Geniş tutmak performansı öldürür. "Hold the lock for the smallest possible region." Wide locks kill performance.
