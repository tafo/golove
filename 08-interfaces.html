1/11

🎭 Niye interface? Why interfaces?

🟦🔺⚪

üçü de farklı tip · üçü de "Alan()" hesaplayabilir three different types · all can compute "Area()"

Interface = "şu davranışlara sahip olan her şey". An interface = "anything that has these behaviours".

Tipi sormak yerine yeteneği sorarsın. Function'ın "Kare mi? Daire mi?" diye umrunda olmaz; "Alan hesaplayabiliyor mu?" yeter. You ask about ability, not type. The function doesn't care "Square? Circle?" — only "can it compute area?".

Nasıl tanımlarız? How do we define one?

2/11

📜 Interface tanımla Define an interface 

type Sekil interface {
    Alan() float64
}

Bir interface = bir metot listesi. An interface = a list of methods.

Alanlar yok, sadece davranışlar. "Bu interface'i karşılayan her tip Alan() float64 metoduna sahip olmalı" der. No fields — only behaviours. It says "any type that satisfies this must have Alan() float64".

"Karşılayan" demek nedir? What does "satisfies" mean?

3/11

🤝 Otomatik karşılama Implicit satisfaction 

type Kare struct { Kenar float64 }

func (k Kare) Alan() float64 {
    return k.Kenar * k.Kenar
}

// Kare artık Sekil interface'ini "karşılar" — kendiliğinden!
// Hiçbir yerde "implements Sekil" demedik.

Doğru metotları yazdıysan, interface otomatik karşılanır. If your type has the right methods, it satisfies the interface automatically.

Java/C#'taki implements keyword'u Go'da yok. Bu yüzden başka birinin yazdığı interface'i de "uygulamış" olabilirsin. No implements keyword like Java/C#. You can satisfy interfaces written by others without knowing.

Function'a interface vermek? Passing interfaces to functions?

4/11

🔌 Function parametresi olarak As a function parameter 

func yazdir(s Sekil) {
    fmt.Printf("alan: %.2f\n", s.Alan())
}

yazdir(Kare{Kenar: 5})        // alan: 25.00
yazdir(Daire{Yaricap: 3})    // alan: 28.27

Tek function, farklı tipler — tek koşul: interface'i karşılıyor olmaları. One function, many types — one condition: they satisfy the interface.

Polimorfizm Go'da böyle yapılır. Generic'lerden önce de bu vardı, hâlâ en yaygın yol. This is how Go does polymorphism. It existed before generics and is still the most common way.

Çoklu tip aynı listede? Many types in one list?

5/11

🎨 Karma slice A mixed slice 

sekiller := []Sekil{
    Kare{Kenar: 5},
    Daire{Yaricap: 3},
    Ucgen{Taban: 4, Yukseklik: 6},
}

for _, s := range sekiller {
    fmt.Println(s.Alan())
}

Farklı tipler, ortak interface, tek slice. Different types, one interface, one slice.

Çağrı sırasında Go hangi gerçek tipin Alan()'ını çağıracağını çözer. Buna dynamic dispatch denir. At call time, Go figures out which concrete Alan() to call. This is dynamic dispatch.

Her şeyi tutabilen interface? An interface that holds anything?

6/11

🌐 any — boş interface any — the empty interface 

var x any          // her şey olabilir

x = 42             // int
x = "merhaba"      // string
x = Kare{Kenar: 3}    // struct

// any === interface{} (eski yazım, aynı şey)

any = "metot listesi yok = her tip karşılar". any = "no method list = every type satisfies".

Go 1.18'de geldi, eskiden interface{} yazılırdı. Çok kullanılırsa kod tip güvenliğini kaybeder — gerektiği yerde kullan. Added in Go 1.18, formerly written interface{}. Overusing it loses type safety — use sparingly.

İçindeki gerçek tipi bulmak? Finding the real type inside?

7/11

🔍 Type assertion Type assertion 

var s Sekil = Kare{Kenar: 5}

k, ok := s.(Kare)    // güvenli — comma ok
if ok {
    fmt.Println(k.Kenar)   // 5
}

k = s.(Kare)         // güvensiz — yanlışsa panic!

Interface'in altındaki gerçek tipe ulaşmak için x.(T). Use x.(T) to reach the concrete type behind an interface.

Daima value, ok := formunu tercih et. Tek değer alırsan ve tip uymuyorsa program panic verir. Always prefer the value, ok := form. The single-value form panics on mismatch.

Birden çok olası tip? Many possible types?

8/11

🎰 Type switch Type switch 

switch v := s.(type) {
case Kare:
    fmt.Println("kare:", v.Kenar)
case Daire:
    fmt.Println("daire:", v.Yaricap)
default:
    fmt.Println("bilmediğim bir şekil")
}

Birden çok tipi kontrol etmek için switch x.(type). For checking many types, use switch x.(type).

Her case'te v o tip olarak gelir — type assertion gerek kalmaz. In each case, v is already that type — no extra assertion needed.

Standart kütüphanedeki ünlüler? Famous stdlib interfaces?

9/11

📚 Stdlib interface'leri Stdlib interfaces

errorError() string · her hata bunu karşılar
fmt.StringerString() string · özel yazdırma
io.ReaderRead([]byte) (int, error) · veri kaynakları
io.WriterWrite([]byte) (int, error) · veri hedefleri
errorError() string · every error satisfies this
fmt.StringerString() string · custom printing
io.ReaderRead([]byte) (int, error) · data sources
io.WriterWrite([]byte) (int, error) · data sinks

Bu dördünü tanırsan stdlib'in yarısını anlarsın. Know these four and half of stdlib makes sense.

Her tipten önce bu interface'leri ara — uyduğunda kodun stdlib ile birleşir. Check these before writing your own — when your type fits, it plugs into stdlib.

Büyük mü küçük mü iyi? Big or small — which is better?

10/11

🐭 Küçük interface'ler Small interfaces win 

// ✓ Tek metot — kullanması ve karşılaması kolay
type Reader interface {
    Read(p []byte) (int, error)
}

// ✗ 12 metot — her şey değişince kırılır
type EveryThing interface { ... }

"Bir interface ne kadar küçükse, o kadar güçlüdür." — Rob Pike "The bigger the interface, the weaker the abstraction." — Rob Pike

Stdlib'in en sevilen interface'leri 1-2 metotludur. Lazım olmadan interface yazma — önce somut tipi yaz, ihtiyaç çıkınca interface çıkar. Stdlib's beloved interfaces have 1–2 methods. Don't define interfaces preemptively — write concrete first, extract when needed.
